                    # CMMI-CM（配置管理）笔记

                    # 项目简介

                    # 2022年中国联通吉林省分公司智能质检服务框架协议（吉林质检项目）

                    本项目是为吉林联通引入智能质检服务，得用DCN网络与服务提供商质检平台对接，提供外呼录音至服务商平台，服务商平台进行智能质检，并将质检结果回传本省外呼系统

                    通过服务商平台，对产品进行质检，通过查看产品信息、用户信息、对话内容、收听通话录音等，来质检是否合格。

                    # 工作步骤

                    # 01 制定项目配置管理计划

                    # 目的

                    用于管理、保存软件生命周期各个配置项，监控各配置项的状态，让小组所有成员能及时了解软件基线的状态和内容。而实现对软件过程的控制，持续改进软件流程，保证软件产品质量、降低风险，同时提高开发团队的工作效率、降低软件开发成本。

                    # CCB 配置控制委员会

                    成员： 高层经理、项目经理、配置人员、质量保证人员

                    # 工作职责

                    配置人员工作： 负责管理公司的配置库、组织财富库。保证公司的文档、代码等资源完整，对项目的阶段产品进行控制，并支持QA 的配置审计工作。
                    配置库： 包含：项目源代码、项目文件文档、项目资料
                    组织财富库： 包含： 公司经验库（总结的工作中的经验教训）、项目规范、项目管理文档、组织方针、培训文件、常用工具（文件上传、日期转换、编码解码）
                    工具： 配置管理服务器（公司机房）+ 配置管理工具（git）

                    # 配置库的权限配置

                    每个成员的权限不同，权限分为：只读、添加、删除、创建目录、签入签出

                    # 备份

                    通过git进行备份

                    1. 配置库的备份：每天1次，操作人：项目经理
                    2. 配置状态的记录和报告备份：2周一次，操作人：配置管理人员

                    # 项目基线管理

                    项目组根据项目所选用的生命周期模型、工作进度的安排制定项目的基线计划，描述项目将要产生的基线、基线的内容要求、大致的时间点、基线标识。项目的基线通常有：计划基线、需求基线、设计基线、编码基线、测试基线、产品基线。

                    基线的内容通常是那些影响下阶段工作的工作产品、文档。

                    # 基线产品

                    需求基线： 需求规格说明书
                    设计基线： 设计说明书
                    测试基线： 系统测试用例
                    试运行基线： 产品源代码V1.1
                    验收基线： 用户手册/产品元赛马V1.2

                    # 如何建立基线/建立流程

                    1. 建立者提出评审申请。
                    2. 项目经理组织相关人进行评审。
                    3. 建立者把评审通过的文档放入“相应的项目库”中。
                    4. 配置管理人员协助QA做好功能审计。
                    5. 审计无误后由配置管理人员填写《配置项纳入申请》提交给CCB审批。
                    6. 审批通过后,配置管理员把配置项放入基线，进行物理审计。填写《基线发布报告》。
                    7. 配置管理员通过邮件告知相关人员配置项已纳入基线。

                    # 基线的变更流程

                    • 进行申请
                    1. 申请人向项目经理提交《需求变更单》。
                    2. 项目经理对变更进行评估，变更较大提交给CCB审批
                    3. CCB审批该申请 分析此变更对项目造成的影响
                    4. 实施变更
                      变更步骤：
                    1. CCB将变更结果提交配置管理员，配置管理员通知项目经理组织实施变更。
                    2. 配置管理员根据变更请求的要求将配置项从配置库中拿出。
                    3. 项目经理指定人员对要变更的配置项进行变更。
                    4. 项目组对更改后的配置项重新进行评审或测试。
                      如果配置项是技术文档，则需要评审（参见《评审过程文件》）。
                      如果配置项是代码需要进行测试。
                    1. 变更的确认
                    1. 项目组将变更后的配置项提交CCB进行确认。
                    2. CCB对变更后的配置项进行确认并批准。
                    3. 配置管理员将确认后的配置项重新放入配置库。
                    4. 配置管理员通知相关的人员，通知方式建议采用邮件方式。
                    5. 配置管理员和质量保证人员要对变更后的配置库进行功能审计和物理审计。

                    # 配置审计

                    基线建前和基线发生变更时进行一次、包含功能审计和物理审计。
                    步骤包含：制定配置计划、配置库的建立、生成《配置状态报告》、配置审计（需求阶段）、配置审计（设计阶段）、配置审计（测试阶段）、配置审计（试运行阶段）、配置审计（验收阶段）

                    # 02 建立《项目配置库目录》

                    根据分类，对文件存放的目录进行整理保存，便于查找。
                    大致包含：开发库资料、前期资料、项目计划、项目需求、项目设计、项目实施、测试文档、培训文档、实施文档、其他文档、项目基线。

                    # 03 建立《配置项状态报告》

                    记录文件的存放位置、文件名称、版本、变更原因、变更次数、变更人等信息。
